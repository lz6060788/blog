# SQLite → PostgreSQL 切换方案（不保留历史数据）

> 状态：**Draft / 待审核**  
> 原则：**立即弃用 SQLite，且不迁移历史数据。**

---

## 1. 决策

本次切换为“重置式切换”：

- 直接停止使用 SQLite。
- 新环境仅使用 PostgreSQL。
- 不做历史数据导入（用户、文章、分类、标签等均不保留）。
- 不做双写、灰度或兼容过渡。

---

## 2. 目标

1. 将线上数据库底座切换为 PostgreSQL。
2. 确保新库结构可支撑当前功能。
3. 将切换步骤标准化，便于一次性执行与审计。

---

## 3. 范围

### 包含

- PostgreSQL 实例准备与连接配置。
- Drizzle PostgreSQL schema/migration 落地。
- 应用数据库驱动切换到 PostgreSQL。
- 发布流程切换与健康检查。
- 清理 SQLite 依赖与运行路径。

### 不包含

- 历史数据迁移与数据对账。
- 双库并行验证。
- 业务模型重构。

---

## 4. 执行步骤（Runbook）

### 阶段 A：准备

- [ ] 创建 PostgreSQL 生产库（建议 v15+）。
- [ ] 创建最小权限账号与网络访问策略。
- [ ] 配置环境变量：
  - `DATABASE_URL=<postgres_dsn>`
- [ ] 在代码中切换为 PostgreSQL 驱动（移除 SQLite 运行分支）。
- [ ] 生成并审查 PostgreSQL 迁移脚本。

### 阶段 B：预发布验证

- [ ] 在测试/预生产环境执行 PostgreSQL migration。
- [ ] 验证所有核心表、索引、外键创建成功。
- [ ] 执行核心功能回归：登录、文章 CRUD、后台设置、AI、音乐模块。
- [ ] 确认应用在“空库初始状态”可正常启动与运行。

### 阶段 C：生产切换

1. 发布维护公告（如需）。
2. 下线当前 SQLite 版本实例。
3. 在生产 PostgreSQL 执行 migration（仅建结构，无数据导入）。
4. 发布 PostgreSQL 版本应用。
5. 执行健康检查与冒烟测试。
6. 验证通过后恢复对外服务。

### 阶段 D：切换后清理

- [ ] 移除 `better-sqlite3` 依赖。
- [ ] 删除 SQLite 历史迁移文件与旧迁移链定义。
- [ ] 移除 SQLite 初始化脚本与生产文档描述。
- [ ] 归档旧 SQLite 文件（仅备份，不再作为回滚数据源）。
- [ ] 连续观察 24~72 小时（错误率、响应时间、连接池）。

---

## 5. 是否应该清除原有迁移记录？（结论：清除）

**结论：数据库层面与代码仓库层面都执行“清空并重建”。**

### 5.1 数据库层面（应清空）

因为不保留历史数据，PostgreSQL 将从空库开始：

- 新库中的迁移记录表（如 `drizzle` 管理表）天然为空；
- 执行新的 PostgreSQL migration 后，记录从 0 开始建立；
- 不需要导入 SQLite 的任何迁移执行记录。

### 5.2 代码仓库层面（直接删除）

按你的最新要求，代码仓库不再保留 SQLite 迁移历史：

- 删除现有 SQLite 迁移文件；
- 重新生成 PostgreSQL baseline migration 作为唯一迁移起点；
- 后续所有迁移均基于 PostgreSQL 链继续追加。

**影响说明：**

- 未来迁移不会受到旧 SQLite 文件干扰；
- 但会失去仓库内的 SQLite 迁移审计链，需接受该取舍。

---

## 6. 回滚策略

由于本方案不保留历史数据，回滚仅针对“服务可用性”，不针对数据恢复：

1. 若 PostgreSQL 版本不可用，回滚到上一个稳定应用版本。
2. 重新启用临时维护页，待问题修复后再次切换。
3. 如需恢复历史内容，需依赖 SQLite 备份离线处理（不在本方案内）。
