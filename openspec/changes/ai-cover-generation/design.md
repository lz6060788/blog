## Context

当前博客系统已实现 AI 摘要生成功能，基于此架构扩展支持 AI 封面图片生成。系统已有的 AI 服务基础设施包括：
- 统一的 AI 客户端管理（支持 DeepSeek、智谱、通义千问等）
- Prompt 模板系统
- AI 调用日志记录
- 数据库模型配置（ai_model_configs、ai_function_mappings）
- 对象存储服务（腾讯云 COS）

封面生成功能将复用这些基础设施，主要扩展点是：
1. 支持图像生成模型（而非仅文本生成）
2. 图像上传到对象存储
3. 封面预览和编辑 UI 组件

## Goals / Non-Goals

**Goals:**
- 根据文章内容自动生成高质量封面图片
- 保持与现有 AI 摘要功能一致的交互体验
- 支持多种图像生成模型（优先使用国内服务商）
- 实现封面生成状态管理和错误处理
- 提供封面预览和手动上传替换功能
- 控制生成成本（频率限制和预算控制）

**Non-Goals:**
- 不实现复杂的图像编辑功能（裁剪、滤镜等）
- 不支持视频或 GIF 生成
- 不实现 AI 图像增强或修复
- 不实现批量封面生成

## Decisions

### 1. 图像生成模型选择

**决策：** 优先使用国内图像生成服务，以 OpenAI DALL-E 作为备选

**理由：**
- 国内服务（通义万相、百度文心一格）更稳定，访问速度更快
- 成本相对较低，支持人民币支付
- 内容审核更符合国内合规要求
- OpenAI DALL-E 作为备选，满足国际化需求

**备选方案：**
- Stable Diffusion：需要额外部署基础设施，运维成本高
- Midjourney：无官方 API，集成复杂度高

### 2. 封面图片存储方案

**决策：** 使用腾讯云 COS 存储，路径规范为 `covers/{year}/{month}/{post_id}_{timestamp}.png`

**理由：**
- 复用现有的 COS 基础设施和文件上传服务
- 与当前文章图片存储保持一致
- 路径设计支持按时间和文章索引，便于管理
- PNG 格式保证质量，适合封面展示

**备选方案：**
- 本地存储：无法支持分布式部署，扩展性差
- CDN 直出：需要额外配置，成本更高

### 3. 封面生成 Prompt 策略

**决策：** 使用结构化 Prompt 模板，包含文章标题、摘要、标签作为上下文

**理由：**
- 结构化 Prompt 保证生成质量的一致性
- 文章标题和摘要提供核心主题信息
- 标签提供风格和类型线索
- Prompt 记录到数据库，便于调试和优化

**模板示例：**
```
Generate a blog post cover image with the following details:
- Title: {title}
- Summary: {summary}
- Tags: {tags}

Requirements:
- Modern, minimalist design
- Professional and clean style
- 16:9 aspect ratio (1200x675)
- High quality, suitable for web display
- Avoid text overlays
```

### 4. 生成状态管理架构

**决策：** 采用与摘要生成一致的状态轮询机制

**理由：**
- 复用现有的前端轮询逻辑
- 统一的用户体验
- 图像生成通常需要 5-15 秒，轮询间隔 3 秒合适

**状态流转：**
```
pending → generating → done/failed/manual
         ↓              ↓
       轮询检查     更新 UI
```

### 5. 频率限制策略

**决策：** 实现两级频率限制（单篇 + 全局）

**单篇限制：** 每篇文章每天最多生成 5 次
- 防止用户重复生成浪费成本
- Redis 记录计数，key: `cover_gen_limit:{post_id}:{date}`

**全局限制：** 全局每分钟最多生成 10 张
- 防止并发请求导致的成本激增
- Redis 记录计数，key: `cover_gen_global:{minute}`

### 6. 数据库扩展设计

**决策：** 在 posts 表新增封面相关字段，而非创建关联表

**理由：**
- 一篇文章只有一个封面，不需要一对多关系
- 查询性能更好，无需 JOIN
- 与现有摘要字段设计保持一致

**新增字段：**
```sql
ALTER TABLE posts ADD COLUMN cover_image_url TEXT;
ALTER TABLE posts ADD COLUMN ai_cover_status VARCHAR(20) DEFAULT 'pending';
ALTER TABLE posts ADD COLUMN ai_cover_generated_at TIMESTAMP;
ALTER TABLE posts ADD COLUMN ai_cover_prompt TEXT;
```

### 7. AI 服务架构扩展

**决策：** 在现有 AI 服务基础上，新增图像生成模块

**目录结构：**
```
server/ai/
├── clients/
│   ├── text-client.ts      # 现有文本生成客户端
│   └── image-client.ts     # 新增图像生成客户端
├── services/
│   ├── summary-service.ts  # 现有摘要生成服务
│   └── cover-service.ts    # 新增封面生成服务
├── prompts/
│   ├── summary-prompt.ts   # 现有摘要 Prompt
│   └── cover-prompt.ts     # 新增封面 Prompt
└── types/
    └── index.ts            # 扩展类型定义
```

### 8. 前端组件设计

**决策：** 创建独立的封面预览组件，集成到文章编辑页面

**组件结构：**
```
components/admin/editor/
├── cover-preview.tsx       # 封面预览组件
│   ├── 展示封面图片
│   ├── 生成 AI 封面按钮
│   ├── 上传封面按钮
│   ├── 删除封面按钮
│   └── 重新生成按钮
```

**状态管理：**
- 复用现有的轮询 Hook（`useAIGenerationStatus`）
- 新增封面相关的状态类型定义

## Risks / Trade-offs

### 风险 1：图像生成质量不稳定
**影响：** 用户对生成的封面不满意，影响使用体验
**缓解措施：**
- 提供重新生成功能，允许用户多次尝试
- 优化 Prompt 模板，提高生成质量
- 支持手动上传作为备选方案

### 风险 2：图像生成 API 成本过高
**影响：** 频繁生成导致运营成本激增
**缓解措施：**
- 实施严格的频率限制
- 记录生成成本，设置预算告警
- 考虑使用成本更低的模型（如通义万相）

### 风险 3：生成内容违规
**影响：** AI 生成不适宜内容的封面图片
**缓解措施：**
- 优先使用有内容审核的国内服务
- 在 Prompt 中明确要求生成健康、专业的内容
- 保留人工审核和删除机制

### 风险 4：图像上传失败
**影响：** AI 生成成功但上传到 COS 失败
**缓解措施：**
- 实现重试机制（最多 3 次）
- 如无法上传，保留 AI 返回的 URL（如有时效性需提示用户）
- 记录详细错误日志，便于排查

### 权衡 1：生成速度 vs 图像质量
**选择：** 优先保证图像质量，接受较长的生成时间
**理由：** 封面是文章的第一印象，质量比速度更重要

### 权衡 2：功能复杂度 vs 开发时间
**选择：** 实现核心功能，暂不支持高级编辑
**理由：** 快速上线验证需求，后续根据用户反馈迭代

## Migration Plan

### 阶段 1：数据库迁移
1. 执行数据库迁移脚本，新增 posts 表字段
2. 更新 TypeScript 类型定义
3. 验证迁移成功

### 阶段 2：后端开发
1. 扩展 AI 服务，新增图像生成客户端
2. 实现封面生成服务
3. 创建封面生成 API 路由
4. 实现频率限制中间件
5. 集成单元测试

### 阶段 3：前端开发
1. 创建封面预览组件
2. 集成到文章编辑页面
3. 实现状态轮询逻辑
4. 新增封面上传功能
5. 更新文章列表和详情页展示封面

### 阶段 4：测试和上线
1. 端到端测试
2. 成本测试和验证
3. 灰度发布
4. 监控和日志

### 回滚策略
- 数据库字段新增不影响现有功能，无需回滚
- API 路由新增，可单独禁用
- 前端组件通过特性开关控制，可快速下线
- 保留手动上传封面功能作为兜底

## Open Questions

1. **图像尺寸标准**
   - 问题：是否支持多种尺寸（如 16:9、4:3、1:1）？
   - 建议：初期仅支持 16:9（1200x675），后续根据需求扩展

2. **封面风格配置**
   - 问题：是否允许用户配置封面风格偏好（如颜色主题、设计风格）？
   - 建议：初期使用统一的默认风格，后续支持个性化配置

3. **历史文章封面迁移**
   - 问题：是否为历史文章批量生成封面？
   - 建议：不批量生成，避免成本浪费，仅对新文章或用户主动触发的文章生成

4. **多模型切换**
   - 问题：用户是否可以选择使用哪个图像生成模型？
   - 建议：初期由管理员统一配置，后续支持用户选择
