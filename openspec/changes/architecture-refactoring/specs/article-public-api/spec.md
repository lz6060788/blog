## MODIFIED Requirements

### Requirement: 获取已发布文章列表
系统应提供从数据库获取已发布文章列表的能力，仅返回 published 状态为 true 的文章，并支持分页和筛选。

#### 场景：获取所有已发布文章（分页）
- **当** 前台页面请求文章列表时
- **那么** 系统应从数据库查询 published=true 的文章
- **并且** 系统应按发布日期倒序排列
- **并且** 系统应支持分页参数（page、limit）
- **并且** 系统应返回分页元数据（total、page、limit、totalPages）
- **并且** 每篇文章应包含 id、title、excerpt、date、category、tags、readTime
- **并且** 查询应使用关联加载避免 N+1 问题

#### 场景：获取所有已发布文章（向后兼容）
- **当** 前台页面未提供分页参数时
- **那么** 系统应返回所有已发布文章
- **并且** 响应格式应与分页格式兼容
- **并且** 响应头应包含弃用警告

#### 场景：按分类筛选文章（分页）
- **当** 用户指定分类筛选时
- **那么** 系统应返回该分类下已发布的文章
- **并且** total 应反映该分类下的文章总数
- **并且** 分页应基于筛选后的结果

#### 场景：按标签筛选文章（分页）
- **当** 用户指定标签筛选时
- **那么** 系统应返回包含该标签的已发布的文章
- **并且** total 应反映包含该标签的文章总数
- **并且** 分页应基于筛选后的结果

#### 场景：组合筛选条件（分页）
- **当** 用户同时指定分类和标签筛选时
- **那么** 系统应返回同时满足两个条件的已发布文章
- **并且** 分页应基于组合筛选的结果

### Requirement: 获取单篇文章详情
系统应根据文章 ID 或 Slug 从数据库获取完整的文章内容，包括作者信息、分类、标签等关联数据，使用优化的查询。

#### 场景：通过 ID 获取文章
- **当** 请求已发布的文章详情时（通过 ID）
- **那么** 系统应返回文章的完整内容
- **并且** 系统应使用关联查询一次获取所有关联数据
- **并且** 系统应包含关联的作者信息
- **并且** 系统应包含关联的分类信息
- **并且** 系统应包含关联的标签列表
- **并且** 查询不应产生 N+1 问题

#### 场景：通过 Slug 获取文章
- **当** 请求已发布的文章详情时（通过 Slug）
- **那么** 系统应返回文章的完整内容
- **并且** 系统应使用关联查询一次获取所有关联数据
- **并且** 查询效率应与 ID 查询相当

#### 场景：请求不存在的文章
- **当** 请求不存在的文章 ID 或 Slug 时
- **那么** 系统应返回 null
- **并且** 页面应显示 404

#### 场景：请求未发布的文章
- **当** 普通用户请求未发布的文章时
- **那么** 系统应返回 null
- **并且** 页面应显示 404

### Requirement: 获取分类列表
系统应提供获取所有可用分类的能力，支持分页（如果分类数量很多）。

#### 场景：获取所有分类
- **当** 前台需要展示分类列表时
- **那么** 系统应返回所有分类
- **并且** 每个分类应包含 id、name、slug、description
- **并且** 如果分类数量超过阈值，支持分页

### Requirement: 获取标签列表
系统应提供获取所有可用标签的能力，支持分页（如果标签数量很多）。

#### 场景：获取所有标签
- **当** 前台需要展示标签列表时
- **那么** 系统应返回所有标签
- **并且** 每个标签应包含 id、name、slug
- **并且** 如果标签数量超过阈值，支持分页

### Requirement: 获取博客作者信息
系统应提供从数据库或设置中获取博客作者信息的能力。

#### 场景：获取作者信息
- **当** 前台需要展示作者信息时
- **那么** 系统应返回作者信息
- **并且** 作者信息应包含 name、avatar、bio、location、email、social

### Requirement: 支持静态页面生成（SSG）
系统应支持在构建时预先生成静态页面，提高访问速度。

#### 场景：生成静态文章页面
- **当** 系统构建时
- **那么** 系统应获取所有已发布文章的 ID
- **并且** 系统应为每篇文章生成静态 HTML
- **并且** 生成过程应使用优化的查询

#### 场景：增量静态再生成（ISR）
- **当** 文章内容更新时
- **那么** 系统应支持增量再生成对应的静态页面

## ADDED Requirements

### Requirement: API 性能监控
系统应监控 API 的查询性能。

#### 场景：记录慢查询
- **当** API 查询时间超过阈值时
- **那么** 系统应记录慢查询日志
- **并且** 日志应包含查询类型、耗时、参数

#### 场景：N+1 查询检测
- **当** 系统检测到潜在的 N+1 查询模式时
- **那么** 系统应记录警告日志
- **并且** 日志应包含问题代码位置

### Requirement: 缓存策略
系统应为前台 API 实现合理的缓存策略。

#### 场景：文章列表缓存
- **当** 请求文章列表时
- **那么** 系统应使用 HTTP 缓存头（Cache-Control）
- **并且** 缓存时间应合理（如 5 分钟）

#### 场景：文章详情缓存
- **当** 请求文章详情时
- **那么** 系统应使用 ETag 或 Last-Modified 头
- **并且** 支持条件请求（304 响应）
