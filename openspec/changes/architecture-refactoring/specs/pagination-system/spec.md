## ADDED Requirements

### Requirement: 统一的分页参数格式
系统应使用统一的分页参数格式。

#### Scenario: 分页参数定义
- **当** 请求列表数据时
- **那么** 应接受 `page` 参数（页码，从 1 开始）
- **并且** 应接受 `limit` 参数（每页数量，默认 20，最大 100）
- **并且** 应接受可选的 `sort` 参数（排序字段和方向）

#### Scenario: 参数默认值
- **当** 未提供 `page` 参数时
- **那么** 系统应使用默认值 1
- **当** 未提供 `limit` 参数时
- **那么** 系统应使用默认值 20
- **当** `limit` 超过最大值时
- **那么** 系统应将其限制为最大值 100

### Requirement: 统一的分页响应格式
系统应使用统一的分页响应格式。

#### Scenario: 分页响应结构
- **当** 返回分页数据时
- **那么** 响应应包含 `data` 字段（数据数组）
- **并且** 应包含 `total` 字段（总记录数）
- **并且** 应包含 `page` 字段（当前页码）
- **并且** 应包含 `limit` 字段（每页数量）
- **并且** 应包含 `totalPages` 字段（总页数）

#### Scenario: 空结果响应
- **当** 查询结果为空时
- **那么** `data` 应为空数组
- **并且** `total` 应为 0
- **并且** `totalPages` 应为 0
- **并且** 不应返回 null

### Requirement: 分页计算逻辑
系统应正确计算分页相关的值。

#### Scenario: 计算总页数
- **当** 计算总页数时
- **那么** 使用公式 `Math.ceil(total / limit)`
- **并且** 当 total 为 0 时，totalPages 应为 0

#### Scenario: 计算 offset
- **当** 执行数据库查询时
- **那么** offset 应使用公式 `(page - 1) * limit`
- **并且** 应处理 page 为 1 的边界情况

#### Scenario: 边界页码处理
- **当** page 超出有效范围时
- **那么** 应返回空数据数组
- **并且** 不应抛出错误
- **并且** total 和 totalPages 应反映实际数据量

### Requirement: 分页参数验证
系统应验证分页参数的有效性。

#### Scenario: 无效的 page 参数
- **当** page 参数小于 1 时
- **那么** 系统应返回 400 错误
- **并且** 错误消息应说明 "page 参数必须大于 0"

#### Scenario: 无效的 limit 参数
- **当** limit 参数小于 1 时
- **那么** 系统应返回 400 错误
- **并且** 错误消息应说明 "limit 参数必须大于 0"

#### Scenario: 参数类型错误
- **当** page 或 limit 不是数字时
- **那么** 系统应返回 400 错误
- **并且** 错误消息应说明参数类型错误

### Requirement: 分页元数据包含导航信息
系统应在分页响应中包含便于前端实现分页导航的信息。

#### Scenario: 包含前后页信息
- **当** 返回分页响应时
- **那么** 响应应包含 `hasPrev` 布尔值（是否有上一页）
- **并且** 应包含 `hasNext` 布尔值（是否有下一页）
- **并且** 应包含 `prevPage` 数字或 null（上一页页码）
- **并且** 应包含 `nextPage` 数字或 null（下一页页码）

#### Scenario: 第一页和最后一页
- **当** 当前页为第一页时
- **那么** `hasPrev` 应为 false
- **并且** `prevPage` 应为 null
- **当** 当前页为最后一页时
- **那么** `hasNext` 应为 false
- **并且** `nextPage` 应为 null

### Requirement: 分页与筛选共存
系统应支持分页与筛选参数同时使用。

#### Scenario: 应用筛选后再分页
- **当** 请求包含筛选和分页参数时
- **那么** 系统应先应用筛选条件
- **并且** total 应反映筛选后的记录数
- **并且** 分页应基于筛选后的结果

#### Scenario: 多个筛选条件
- **当** 请求包含多个筛选条件时
- **那么** 系统应应用所有筛选条件
- **并且** 分页应基于组合筛选的结果

### Requirement: API 分页向后兼容
系统应在过渡期保持分页 API 的向后兼容性。

#### Scenario: 未提供分页参数时返回所有数据
- **当** 请求未包含 page 和 limit 参数时
- **那么** 系统应返回所有符合条件的数据
- **并且** 响应格式应与分页格式兼容
- **并且** 应在响应头中包含弃用警告

#### Scenario: 显式请求所有数据
- **当** 请求设置 limit 为 0 或特殊值（如 "all"）时
- **那么** 系统应返回所有符合条件的数据
- **并且** 应记录弃用日志

### Requirement: Server Actions 分页支持
系统应在 Server Actions 中支持分页参数。

#### Scenario: Server Action 接受分页参数
- **当** Server Action 需要返回列表数据时
- **那么** Action 应接受 `page` 和 `limit` 参数
- **并且** 应返回标准分页格式的结果

#### Scenario: 表单提交后的分页状态
- **当** 表单提交后需要展示列表时
- **那么** 分页状态应重置为第一页
- **并且** 前端应正确处理状态重置

### Requirement: 分页性能优化
系统应优化分页查询的性能。

#### Scenario: 使用 COUNT 查询总数
- **当** 计算总记录数时
- **那么** 系统应使用独立的 COUNT 查询
- **并且** 不应获取所有记录来计算总数

#### Scenario: 大偏移量性能优化
- **当** page 很大时（如超过 100）
- **那么** 系统应记录性能日志
- **并且** 考虑使用游标分页（可选优化）

### Requirement: 分页响应示例
系统应提供清晰分页响应示例。

#### Scenario: 标准分页响应
- **当** 请求第 2 页，每页 20 条记录时
- **那么** 响应格式应为：
```json
{
  "data": [...], // 20 条记录
  "total": 156,
  "page": 2,
  "limit": 20,
  "totalPages": 8,
  "hasPrev": true,
  "hasNext": true,
  "prevPage": 1,
  "nextPage": 3
}
```
