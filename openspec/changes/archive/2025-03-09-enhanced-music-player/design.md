## Context

当前博客系统基于 Next.js 14，使用 Drizzle ORM + SQLite，已集成腾讯云 COS 和 AI 服务。项目采用前后端一体的架构，拥有完善的用户认证和管理后台。现有技术栈包括 Radix UI 组件库、Framer Motion 动画库、Tailwind CSS 样式系统。

音乐播放功能是全新特性，需要在现有架构基础上无缝集成，同时确保不影响现有功能（页面切换、路由、性能）。

## Goals / Non-Goals

**Goals:**
- 构建全站统一的全局音乐播放器，页面切换不中断播放
- 实现精美、现代化的 UI 设计，采用拟态玻璃风格和极简主义
- 支持唱片旋转动画和粒子浮动可视化效果
- 提供完整的音乐管理后台，支持自托管存储
- 优先保证 UI/UX 质量，第一阶段专注 UI 原型开发

**Non-Goals:**
- 不涉及社交分享功能（评论、推荐等）
- 不做音乐版权管理或 DRM
- 不支持外部音乐平台集成（Spotify、网易云等）
- 初期不支持音乐下载功能

## Decisions

### 1. 状态管理选择：Zustand

**决策：** 使用 Zustand 管理播放器全局状态

**理由：**
- 轻量级（~1KB），无需 Context Provider 包装
- TypeScript 支持优秀，API 简洁直观
- 支持状态持久化中间件，方便保存播放偏好
- 比 Redux 更简单，比 Context 性能更好

**替代方案考虑：**
- React Context：会导致不必要的重渲染，且需要 Provider 嵌套
- Redux：过于重量级，对于单一播放器功能过度设计
- Jotai/Valtio：也是不错的选项，但 Zustand 生态更成熟

### 2. 播放器架构：单例模式

**决策：** 在 `layout.tsx` 中挂载单例播放器组件

**理由：**
- 确保全站只有一个播放器实例
- 页面路由切换时播放器不卸载
- 音频元素（Audio Element）可以持续播放
- 状态管理更加集中和可预测

**实现要点：**
- 使用 `useMemo` 确保 Zustand store 只初始化一次
- 音频元素作为 store 的一部分，不受组件生命周期影响
- 通过 `sessionStorage` 持久化当前播放状态

### 3. 音频可视化：Canvas 粒子效果

**决策：** 使用 Canvas API 实现粒子浮动效果，不改变粒子大小

**理由：**
- 粒子大小固定（3-4px），仅调整透明度（0.3-0.8）和数量（20-50）
- 使用 Web Audio API 的 `AnalyserNode` 获取音频频率数据
- Canvas 性能优于 DOM 动画，支持更多粒子
- 粒子颜色可自定义，默认跟随主题色

**替代方案考虑：**
- CSS 动画：无法响应音频频率数据，效果单调
- Three.js：过度设计，增加不必要的复杂度和包体积
- WebGL：性能最优但开发成本高，对于简单粒子效果不必要

### 4. UI 样式：拟态玻璃风格

**决策：** 采用 Glassmorphism 设计语言

**核心样式：**
```css
background: rgba(255, 255, 255, 0.1);
backdrop-filter: blur(20px);
border: 1px solid rgba(255, 255, 255, 0.18);
box-shadow: 0 8px 32px rgba(0, 0, 0, 0.1);
border-radius: 16-20px;
```

**理由：**
- 现代感强，与深色/浅色主题都能很好融合
- 模糊背景不会完全遮挡页面内容
- 符合用户"现代化、简洁主义"的要求

### 5. 音乐存储：COS 自托管

**决策：** 音频文件存 COS，元数据存 SQLite

**存储策略：**
- 音频文件路径：`music/{year}/{month}/{songId}.{ext}`
- 封面图片：`music/covers/{songId}.webp`
- 使用临时签名 URL（5分钟有效期），避免长期暴露
- 支持断点续传大文件

**理由：**
- 充分利用现有的腾讯云 COS 基础设施
- 数据库存储轻量级元数据，查询速度快
- COS CDN 加速，播放体验好
- 自托管避免第三方 API 限制和版权问题

### 6. 开发优先级：UI 优先策略

**决策：** 第一阶段创建测试页面 `/music-player-demo`，纯静态 UI 调试

**开发阶段：**
1. **阶段一（最高优先级）：** UI 原型开发
   - 创建测试页面，使用 mock 数据
   - 实现所有 UI 组件和动画效果
   - 提供粒子效果参数实时调节面板
   - 响应式测试（移动端/桌面端）

2. **阶段二：** 播放器核心逻辑
   - 集成 Zustand 全局状态
   - 实现音频播放控制
   - 连接真实音频文件

3. **阶段三：** 后端集成
   - API 开发
   - COS 上传集成
   - 数据库持久化

**理由：**
- UI 是用户体验的核心，提前发现问题成本低
- 纯静态开发速度快，迭代灵活
- 确保 UI/UX 质量后再投入后端开发
- 避免后端改动导致 UI 重复调整

## Risks / Trade-offs

### 风险 1：页面切换导致音频中断

**风险：** Next.js 路由切换可能导致 Audio 元素重新初始化

**缓解措施：**
- 将 Audio 元素存储在 Zustand store 中，不依赖组件生命周期
- 使用 `useMemo` 和 `useEffect` 确保 Audio 实例只创建一次
- 测试所有路由切换场景，确保播放连续性

### 风险 2：COS CORS 配置问题

**风险：** 浏览器直接访问 COS 可能遇到 CORS 错误

**缓解措施：**
- 在 COS 存储桶配置中添加正确的 CORS 规则
- 必要时通过服务端代理转发音频请求
- 提供友好的错误提示和重试机制

### 风险 3：音频文件过大导致上传慢

**风险：** 高品质音频文件可能几十 MB，上传耗时

**缓解措施：**
- 实现断点续传功能
- 显示上传进度条
- 前端压缩音频（可选，使用 Web Audio API）
- 设置合理的文件大小限制（如 50MB）

### 权衡 1：粒子效果 vs 性能

**权衡：** 更多粒子 = 更好视觉效果，但可能影响低端设备性能

**决策：**
- 默认 30 个粒子，性能友好
- 提供设置选项让用户调整粒子数量
- 使用 `requestAnimationFrame` 优化渲染
- 考虑在低端设备上禁用粒子效果

### 权衡 2：玻璃拟态 vs 可读性

**权衡：** 背景模糊可能影响内容可读性

**决策：**
- 调整背景透明度和模糊度，找到平衡点
- 确保文本对比度符合 WCAG 标准
- 提供关闭背景模糊的选项

## Migration Plan

### 部署步骤

1. **UI 原型发布（开发环境）**
   - 部署测试页面 `/music-player-demo`
   - 收集 UI/UX 反馈并迭代

2. **数据库迁移**
   ```sql
   -- 执行 Drizzle 迁移脚本
   npm run db:generate
   npm run db:migrate
   ```

3. **功能发布（灰度）**
   - 先启用音乐管理后台，上传测试内容
   - 再启用前端播放器组件
   - 监控性能指标和错误日志

4. **全量发布**
   - 确认无重大问题后全量上线
   - 更新用户文档

### 回滚策略

- **数据库回滚：** 使用 Drizzle 回滚功能，保留备份
- **功能回滚：** 通过功能开关（Feature Flag）快速关闭播放器
- **前端回滚：** 回滚到上一个 commit，不影响现有功能

## Open Questions

1. **音频格式支持：** 是否需要支持 FLAC、APE 等无损格式？（初期建议只支持 MP3、AAC、OGG）
2. **歌词来源：** 是否集成第三方歌词 API？（初期建议手动上传或内嵌 LRC）
3. **移动端体验：** 播放器在移动端是否需要不同的布局？（需要响应式测试确认）
4. **播放模式：** 是否支持随机播放、单曲循环等模式？（建议基础版支持）
