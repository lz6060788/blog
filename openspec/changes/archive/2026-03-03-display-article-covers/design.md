## Context

当前博客系统已具备完整的封面功能后端支持（AI 生成和手动上传），数据库中 `posts` 表已包含 `coverImageUrl` 字段，但在前端页面中尚未展示。

**现有组件**：
- `components/admin/ai/cover-preview.tsx` - 管理端封面预览和编辑组件
- Next.js Image 组件 - 用于优化的图片加载

**技术约束**：
- 使用 Next.js App Router 和 SSR
- 遵循现有的主题系统和设计规范
- 需要保持良好的性能和用户体验

## Goals / Non-Goals

**Goals:**
- 在四个关键位置展示文章封面，提升视觉吸引力
- 实现响应式设计，适配各种屏幕尺寸
- 优化图片加载性能（使用 Next.js Image 优化）
- 提供优雅的降级体验（无封面时的占位符）
- 保持与现有设计系统的一致性

**Non-Goals:**
- 不修改封面生成或上传功能（已存在）
- 不改变数据库结构
- 不添加新的 API 端点
- 不实现封面裁剪功能（使用原图）

## Decisions

### 1. 封面尺寸和比例

**决策**：统一使用 16:9 比例，不同页面使用不同的显示尺寸

- **管理端列表**：160x90px 小尺寸缩略图，16:9 比例
- **用户端列表**：16:9 比例卡片封面，宽度自适应
- **归档页面**：16:9 比例中等尺寸封面
- **文章详情页**：16:9 比例大尺寸头图，全宽

**理由**：
- 统一的比例简化了封面生成和裁剪流程
- 16:9 是最常见的高清视频和图片比例，用户体验好
- 不同尺寸通过宽度和容器控制，保持视觉一致性
- 管理端使用较小尺寸但仍保持 16:9，不影响表格高度

**替代方案**：为不同位置使用不同比例（1:1、4:3、16:9）
- **问题**：需要生成多种尺寸的封面，增加复杂度和存储成本

### 2. 图片加载策略

**决策**：使用 Next.js Image 组件，配置以下特性

- 启用 `priority` 属性加载首屏图片
- 使用 `fill` 模式实现响应式布局
- 配置 `blurDataURL` 实现加载占位符
- 设置合理的 `sizes` 属性优化响应式加载

**理由**：
- Next.js Image 自动优化图片格式（WebP/AVIF）
- 提供内置的懒加载和响应式支持
- 符合 Next.js 最佳实践

### 3. 降级处理

**决策**：为无封面文章提供占位符设计

- **管理端**：显示图标占位符（ImageIcon）
- **用户端列表**：显示默认渐变色背景 + 图标
- **归档页面**：显示简洁的卡片样式
- **详情页**：显示简洁的顶部装饰或跳过封面区域

**理由**：
- 保持页面美观和一致性
- 避免布局破坏
- 提供视觉提示

### 4. 组件复用策略

**决策**：创建新的封面展示组件，不完全复用现有的 `cover-preview.tsx`

- 创建 `components/article/article-cover.tsx` - 纯展示组件
- 创建 `components/article/article-cover-card.tsx` - 卡片组件（列表用）
- 保留 `components/admin/ai/cover-preview.tsx` 用于管理端编辑

**理由**：
- `cover-preview.tsx` 包含编辑逻辑和状态管理，不适合纯展示
- 新组件更轻量，职责单一
- 管理端和展示端的需求差异较大

**替代方案**：扩展现有组件
- **问题**：会增加组件复杂度，违反单一职责原则

### 5. 响应式布局

**决策**：使用 Tailwind CSS 的响应式类实现断点布局

- 移动端（< 768px）：单列布局，封面全宽
- 平板端（768px - 1024px）：两列布局
- 桌面端（≥ 1024px）：三列或四列布局

**理由**：
- 符合现有的响应式设计模式
- Tailwind 已在项目中使用

## Risks / Trade-offs

**风险 1：图片加载性能**
- **问题**：大量封面可能影响页面加载速度
- **缓解**：使用 Next.js Image 优化，懒加载非首屏图片，配置图片尺寸限制

**风险 2：图片尺寸不一致**
- **问题**：用户上传的封面尺寸和比例各异
- **缓解**：使用 `object-fit: cover` 裁剪显示，CSS 控制显示尺寸

**风险 3：移动端流量消耗**
- **问题**：封面图片会增加用户流量消耗
- **缓解**：移动端加载较小尺寸图片，提供数据节省模式选项（未来）

**风险 4：管理端列数过多**
- **问题**：添加封面列后管理端列表可能过宽
- **缓解**：使用响应式表格，移动端隐藏次要列，封面列始终显示

## Migration Plan

**阶段 1：组件开发**
1. 创建基础封面展示组件
2. 创建卡片封面组件
3. 添加单元测试

**阶段 2：页面集成**
1. 集成到管理端文章列表
2. 集成到用户端首页
3. 集成到归档页面
4. 集成到文章详情页

**阶段 3：优化和测试**
1. 性能优化（图片加载、懒加载）
2. 响应式测试
3. 浏览器兼容性测试
4. 无封面场景测试

**回滚策略**：
- 所有变更都是新增组件和修改布局
- 可以通过移除封面组件快速回滚
- 不涉及数据库变更，回滚无风险

## Open Questions

1. **默认封面设计**：是否需要设计一套默认封面图案（如渐变色、几何图形）？
   - **倾向**：先使用简洁的图标占位符，后续迭代添加默认封面设计

2. **封面 alt 文本**：自动生成规则是什么？
   - **建议**：使用文章标题作为 alt 文本，或"文章封面"作为默认

3. **历史文章处理**：是否为现有历史文章批量生成封面？
   - **建议**：不批量生成，让用户按需使用 AI 生成功能

4. **图片 CDN 配置**：是否需要配置 CDN 加速封面图片？
   - **评估**：当前使用腾讯云 COS，已支持 CDN，无需额外配置
