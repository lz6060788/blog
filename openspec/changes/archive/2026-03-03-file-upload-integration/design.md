## 背景信息

当前博客系统基于 Next.js 架构，后端使用 API Routes 处理服务器端逻辑。系统已集成腾讯云 AI 服务用于文章摘要生成，现在需要扩展腾讯云集成以支持对象存储（COS）服务。

**技术栈**：
- 前端：Next.js 14+ (App Router)、React、TypeScript、Tailwind CSS
- 后端：Next.js API Routes
- 编辑器：Cherry Markdown
- 数据库：Prisma ORM
- 云服务：腾讯云 COS

**约束条件**：
- 需要确保向后兼容，不破坏现有功能
- 文件上传必须安全，防止恶意文件上传
- COS 配置信息需要安全存储（环境变量）
- 上传功能需要良好的用户体验（进度反馈、错误处理）

## 目标与非目标

**目标：**
- 提供稳定、安全的文件上传功能，支持图片和文档
- 集成腾讯云 COS 作为文件存储后端
- 提供统一的前端上传接口，支持多种使用场景
- 集成 Cherry Markdown 编辑器的图片上传功能
- 实现文件类型和大小验证，确保系统安全

**非目标：**
- 不支持视频文件上传（可后续扩展）
- 不实现文件管理界面（仅提供上传和删除 API）
- 不实现图片处理功能（裁剪、缩放、水印等）
- 不支持批量文件上传管理界面

## 技术决策

### 1. 腾讯云 COS SDK 选择

**决策**: 使用 `cos-nodejs-sdk-v5` 官方 SDK

**理由**:
- 官方维护，稳定性高，文档完善
- 支持丰富的功能（上传、删除、签名 URL 等）
- 与腾讯云服务深度集成，性能优化
- 社区活跃，问题解决快

**替代方案**:
- 使用通用 S3 SDK：需要适配腾讯云，配置复杂
- 自己实现 HTTP 请求：维护成本高，易出错

### 2. 文件存储结构

**决策**: 使用扁平化目录结构，基于时间戳生成唯一文件名

**格式**: `{year}/{month}/{timestamp}-{random}.{ext}`

**示例**: `2026/03/1678123456-abc123.jpg`

**理由**:
- 避免文件名冲突（用户上传同名文件）
- 按时间分片，便于管理和备份
- 保持 URL 简洁，便于缓存
- 避免深层目录结构，提高访问性能

**替代方案**:
- 使用 UUID 文件名：URL 可读性差
- 使用用户 ID 分组：隐私问题，URL 可预测

### 3. 文件元数据存储

**决策**: 在数据库中创建 `FileUpload` 模型存储元数据

**字段**:
- `id`: 主键
- `key`: COS 对象键
- `filename`: 原始文件名
- `size`: 文件大小（字节）
- `mimeType`: MIME 类型
- `uploaderId`: 上传者用户 ID
- `createdAt`: 上传时间

**理由**:
- 便于查询和管理用户上传的文件
- 支持权限验证（用户只能删除自己的文件）
- 便于统计和审计
- 支持未来的文件管理功能

**替代方案**:
- 仅依赖 COS 元数据：查询效率低，无法关联用户
- 不存储元数据：无法实现权限控制和审计

### 4. 前端上传 Hook 设计

**决策**: 创建 `useFileUpload` Hook，封装上传逻辑

**API**:
```typescript
const { upload, isUploading, progress, error, reset } = useFileUpload()
```

**理由**:
- 提供统一的上传接口，代码复用性高
- 管理上传状态（进度、错误），简化组件逻辑
- 易于测试和维护
- 支持多种使用场景（编辑器、独立上传组件等）

**替代方案**:
- 直接在组件中调用 API：代码重复，状态管理分散
- 使用第三方上传库：增加依赖，可能过度设计

### 5. Cherry Markdown 集成方式

**决策**: 配置 Cherry Markdown 的 `upload` 选项，使用自定义上传函数

**实现**:
```typescript
const Cherry = require('cherry-markdown')
const cherry = new Cherry({
  upload: async (file) => {
    const result = await uploadFile(file)
    return result.url
  }
})
```

**理由**:
- 利用编辑器内置的上传功能，支持拖拽、粘贴等
- 配置简单，侵入性小
- 用户体验好（自动上传、自动插入）

**替代方案**:
- 监听编辑器事件，手动处理上传：复杂度高，兼容性问题
- 禁用编辑器上传，使用独立上传组件：用户体验差

### 6. 文件类型和大小限制

**决策**: 前后端双重验证，白名单机制

**图片类型**: JPG, PNG, GIF, WEBP, SVG
**文档类型**: PDF, DOC, DOCX, TXT, MD
**大小限制**: 5MB

**理由**:
- 白名单机制更安全，防止恶意文件上传
- 前端验证提升用户体验（快速反馈）
- 后端验证确保安全性（绕过前端也不行）
- 5MB 限制平衡用户需求和存储成本

**替代方案**:
- 黑名单机制：容易遗漏新的恶意文件类型
- 仅前端验证：不安全，可被绕过

### 7. 上传进度追踪

**决策**: 使用 Axios 的 `onUploadProgress` 回调追踪进度

**实现**:
```typescript
axios.post('/api/upload', formData, {
  onUploadProgress: (progressEvent) => {
    const percentCompleted = Math.round(
      (progressEvent.loaded * 100) / progressEvent.total
    )
    // 更新进度
  }
})
```

**理由**:
- Axios 是项目已使用的 HTTP 客户端
- 提供原生进度追踪支持
- 性能好，兼容性高

**替代方案**:
- 使用 XMLHttpRequest：代码复杂，不易维护
- 使用第三方上传库：增加依赖

## 风险与权衡

### 风险 1: COS 配置泄露
**风险**: 环境变量中的 COS 密钥可能泄露，导致安全问题

**缓解措施**:
- 使用 `.env.local` 文件（不提交到 Git）
- 在 `.gitignore` 中排除环境变量文件
- 定期轮换密钥
- 使用最小权限原则配置 COS 密钥
- 考虑使用临时密钥（STS）进行上传

### 风险 2: 恶意文件上传
**风险**: 攻击者可能上传恶意文件（伪装成图片的脚本）

**缓解措施**:
- 严格的文件类型白名单
- 验证文件 MIME 类型和扩展名
- 限制文件大小
- 后端验证文件魔数（file signature）
- 定期扫描存储的文件
- 考虑使用病毒扫描服务

### 风险 3: 存储成本失控
**风险**: 用户大量上传文件导致存储成本激增

**缓解措施**:
- 限制单用户上传总量（配额）
- 实施文件清理策略（删除未引用的文件）
- 定期审计存储使用情况
- 考虑使用 CDN 回源流量优化

### 风险 4: 上传性能问题
**风险**: 大文件上传或并发上传导致性能问题

**缓解措施**:
- 限制文件大小（5MB）
- 实施上传速率限制
- 考虑使用分片上传（对于大文件）
- 使用 CDN 加速上传
- 监控上传性能指标

### 权衡 1: 功能复杂度 vs 开发时间
**权衡**: 完整的文件管理功能需要大量开发时间

**决策**: 先实现核心上传和删除功能，后续迭代添加管理界面

### 权衡 2: 用户体验 vs 安全性
**权衡**: 严格的验证可能影响用户体验

**决策**: 优先保证安全性，通过良好的错误提示提升体验

## 迁移计划

### 部署步骤

1. **准备阶段**
   - 在腾讯云控制台创建 COS Bucket
   - 配置 Bucket 权限（公共读私有写）
   - 获取 COS 密钥（SecretId、SecretKey）
   - 配置 CORS 规则允许前端域名访问

2. **环境配置**
   - 添加环境变量到 `.env.local`:
     ```
     COS_SECRET_ID=your_secret_id
     COS_SECRET_KEY=your_secret_key
     COS_BUCKET=your_bucket_name
     COS_REGION=ap-guangzhou
     ```
   - 在生产环境配置相同的环境变量

3. **数据库迁移**
   - 创建 `FileUpload` 模型
   - 运行 Prisma 迁移:
     ```bash
     npx prisma migrate dev --name add_file_upload
     ```

4. **依赖安装**
   - 安装 COS SDK:
     ```bash
     npm install cos-nodejs-sdk-v5
     ```

5. **代码部署**
   - 部署后端上传服务
   - 部署前端上传 Hook 和 API
   - 部署 Cherry Markdown 集成

6. **测试验证**
   - 测试图片上传功能
   - 测试文档上传功能
   - 测试编辑器集成
   - 测试错误处理

### 回滚策略

如果部署后出现问题，可以按以下步骤回滚：

1. **快速回滚**：禁用上传功能（通过功能开关）
2. **代码回滚**：回退到上一个稳定版本
3. **数据清理**：删除已上传的文件和数据库记录
4. **配置回滚**：移除 COS 环境变量

## 待解决问题

1. **是否需要支持文件版本管理？**
   - 当前设计不支持文件版本覆盖
   - 同名文件会生成新的唯一文件名
   - 如需支持，需要额外设计版本管理机制

2. **是否需要实现图片压缩？**
   - 当前设计直接上传原始文件
   - 图片压缩可以减少存储成本和传输时间
   - 需要权衡开发成本和收益

3. **是否需要实现文件过期机制？**
   - 当前设计文件永久存储
   - 可以考虑设置过期时间，自动清理旧文件
   - 需要根据业务需求决定

4. **是否需要支持私有文件？**
   - 当前设计所有文件都是公开访问
   - 如需支持私有文件，需要实现签名 URL 机制
   - 需要额外的权限管理

5. **如何处理未引用的文件？**
   - 用户可能上传文件但未在文章中使用
   - 需要定期清理未引用的文件
   - 可以实现文件引用追踪机制
