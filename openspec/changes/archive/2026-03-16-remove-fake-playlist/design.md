## Context

当前系统在音乐列表为空时，前端会返回一个虚假/示例播放列表。这通常是因为：
1. 前端为了演示效果硬编码了示例数据
2. API 在无数据时返回了示例数据而非空数组

这会给用户造成误导，让用户以为有音乐可以播放，但点击后无法播放或出错。

## Goals / Non-Goals

**Goals:**
- 移除虚假/示例播放列表
- 当音乐列表为空时，返回空数组或显示明确的空状态
- 改善用户体验，让用户清楚当前没有可播放的音乐

**Non-Goals:**
- 不修改播放器的 UI 布局
- 不添加"默认音乐"功能

## Decisions

### 1. API 层面返回空数组
- 当数据库中没有音乐时，API 应返回空数组 `[]` 而非示例数据
- 前端根据空数组显示空状态

### 2. 前端空状态处理
- 当播放列表为空时，显示友好的空状态提示（如"暂无音乐"）
- 可选：在空状态时显示如何添加音乐的指引（指向管理后台）

### 3. 定位虚假数据来源
- 需要检查前端组件中是否有硬编码的示例数据
- 需要检查 API 是否在无数据时返回了默认数据

## Risks / Trade-offs

- **风险**: 如果用户习惯了看到示例列表，可能会误以为系统出故障
  - **缓解**: 空状态提示应该友好且清晰，解释"管理员还未添加音乐"

- **风险**: 开发/演示环境可能需要示例数据来测试播放器
  - **缓解**: 如果需要，可以添加一个"演示模式"开关，仅在特定环境下启用

## Migration Plan

1. 定位虚假数据的来源（前端组件或 API）
2. 移除硬编码的示例数据
3. 确保空列表时返回空数组
4. 添加空状态 UI 提示
5. 测试：验证空音乐列表时显示正确状态
6. 部署

**回滚策略**: 恢复原有的数据返回逻辑
