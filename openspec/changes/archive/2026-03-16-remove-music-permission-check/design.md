## Context

当前音乐播放列表 API (`/api/music/playlist`) 使用了权限校验中间件（可能是 `requireAuth` 或类似），限制只有登录用户或特定权限的用户才能访问音乐列表。这对于博客网站的音乐播放器来说过于严格，因为音乐播放器是公开的网站功能，应该让所有访问者都能听到。

当前状态：
- 音乐 API 有权限检查
- 前端需要处理 401/403 错误
- 未登录用户无法获取播放列表

## Goals / Non-Goals

**Goals:**
- 移除音乐播放列表 API 的权限校验，使所有用户都能访问
- 简化前端代码，移除不必要的权限错误处理
- 保持音乐管理后台的权限控制（仅管理员可以上传/删除音乐）

**Non-Goals:**
- 不修改音乐管理功能（上传、删除、编辑）的权限控制
- 不修改其他 API 的权限策略

## Decisions

### 1. 移除播放列表 API 的权限中间件
- 移除 `/api/music/playlist` 端点的 `requireAuth` 或类似中间件
- 保留 `/api/music/admin/*` 端点的权限检查（管理功能）

### 2. 简化前端错误处理
- 移除前端对音乐 API 401/403 错误的特殊处理
- 只保留网络错误和数据格式错误的处理

### 3. 安全考虑
- 音乐数据本身是公开内容（类似博客文章），移除权限是合理的
- 如果需要限制访问，应在 CDN 层面或网站层面做访问控制，而非在 API 层

## Risks / Trade-offs

- **风险**: 如果未来需要针对特定地区限制音乐访问，需要重新添加权限检查
  - **缓解**: 在网站层面实现 IP 或地区限制，而非 API 权限

- **风险**: 可能被滥用进行大量请求
  - **缓解**: 添加速率限制（rate limiting），但不使用身份验证

## Migration Plan

1. 修改后端 API 路由，移除播放列表端点的权限中间件
2. 前端移除权限错误处理逻辑
3. 测试：验证未登录用户可以获取和播放音乐
4. 部署

**回滚策略**: 如果出现问题，可以快速恢复权限中间件
